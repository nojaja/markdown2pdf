# markdown2pdf CLI 設計書（v1.00）

## 1. 目的
`markdown-it-mermaid` と `commander` を用いて、Mermaid記法を含むMarkdownをPDFへ変換するCLIアプリを設計する。

本書は設計のみを対象とし、実装（製造）は行わない。

## 2. スコープ
- 対象: CLIアプリの要件、インターフェース、内部構成、入出力ルール、エラー方針
- 非対象: 実装コード作成、パッケージ導入、テスト実装、リリース作業

## 3. 採用技術
- CLI引数処理: `commander`
- Markdown変換: `markdown-it`
- Mermaid対応: `markdown-it-mermaid`
- PDF生成方式（初版採用）: ヘッドレスブラウザでHTML描画後にPDF化

## 4. CLI仕様

### 4.1 コマンド
- コマンド名: `md2pdf`

### 4.2 オプション
- `--input <path>`: 入力（ファイル/フォルダ）
- `--output <path>`: 出力（ファイル/フォルダ）
- `--css <path>`: 追加CSS（任意）
- `--format <A4|Letter>`: 用紙サイズ
- `--landscape`: 横向き
- `--margin-top <value>`
- `--margin-right <value>`
- `--margin-bottom <value>`
- `--margin-left <value>`
- `--scale <0.1-2>`
- `--config <path>`: 設定ファイル（JSON）
- `--timeout <ms>`: Mermaid描画待ちタイムアウト
- `--debug`: 中間成果物保存・詳細ログ
- `--dry-run`: 実変換なしで対象/出力先のみ表示

### 4.3 AIエージェント利用性向上CR
本CLIは Claude Code や GitHub Copilot 等のAIエージェント利用を想定し、以下を追加する。

1. helpの機械可読化
  - `--help-format <text|json>` を追加する。
  - `json` 指定時は、全オプション、型、必須性、既定値、相互制約をJSONで返す。
  - 人間向け表示（text）と同一情報を保持し、差分を作らない。

2. 実行結果の機械可読化
  - `--report-format <text|json>` を追加する。
  - `json` 指定時は以下を返す。
    - `summary`: 対象件数/成功件数/失敗件数/終了コード
    - `outputs`: 入力ファイルと出力ファイルの対応
    - `failures`: 失敗ファイル、失敗種別、詳細理由
  - text形式は従来どおり人間可読ログとする。

3. 実行計画の事前取得
  - `--plan` を追加する。
  - 変換は実行せず、対象ファイル列挙と最終出力先マッピングのみ返す。
  - `--plan --report-format json` の併用で、エージェントが実行前検証できる。

4. 終了コードの厳格化
  - 既定の終了コード定義を維持しつつ、`--strict-exit-code` を追加する。
  - 部分成功時（通常は`1`）の扱いを固定し、曖昧な成功判定を避ける。

5. エラーメッセージの構造化
  - 失敗種別を固定語彙で出力する（`INPUT_VALIDATION`, `RENDER_TIMEOUT`, `PDF_WRITE`, `UNEXPECTED`）。
  - text形式でも先頭に失敗種別コードを付与し、エージェントの解析を容易にする。

6. 安全実行ガード
  - `--yes` がない場合、破壊的動作を伴う将来拡張（上書き/削除）を実行しない設計とする。
  - 初版で破壊的動作がない場合も、予約オプションとして仕様化する。

7. 冪等・再実行性
  - 同一入力・同一オプション時に同一出力パスを決定することを保証する。
  - `--report-format json` の出力スキーマを固定し、将来変更時はバージョン付きで管理する。

8. LLM向けヘルプ文最適化
  - 各オプション説明に「期待入力例」「禁止パターン」を1行で含める。
  - help本文末尾に「代表的な実行例（単一/一括/plan/json）」を必ず表示する。

### 4.4 入力対象の探索
- `--input` がファイルの場合:
  - そのファイル1件を対象とする。
- `--input` がフォルダの場合:
  - 配下の `*.md` を再帰探索して対象とする（デフォルトON）。
  - 対象リストはパス昇順で安定ソートする。
  - 対象0件はエラー。

## 5. 出力パス解決ルール

### 5.1 単一入力（input=ファイル）
1. output未指定:
   - 入力と同階層へ `入力ベース名.pdf`
2. output=ファイル:
   - 指定ファイルへ出力
3. output=フォルダ:
   - `output/入力ベース名.pdf`

### 5.2 複数入力（input=フォルダ）
1. output未指定:
   - `input/_pdf/` を作成し、各PDFを出力
2. output=フォルダ:
   - 指定フォルダ配下に出力
3. output=ファイル:
   - 指定ファイルの親ディレクトリ配下に出力

### 5.3 複数入力時の命名・配置（確定）
重複有無に関わらず、出力先には `input` フォルダ配下と同様の相対フォルダ構造を再現してPDFを配置する。

例:
- 入力: `docs/a/intro.md`, `docs/b/intro.md`
- output=フォルダ `dist` の場合:
  - `dist/a/intro.pdf`
  - `dist/b/intro.pdf`
- output=ファイル `dist/result.pdf` の場合:
  - `dist/result_a/intro.pdf`
  - `dist/result_b/intro.pdf`

補足:
- output=ファイル時は、`出力ファイル名(拡張子なし)_<input直下1階層>` をトップフォルダ名として使う。
- input直下直ファイル（例: `docs/root.md`）は `dist/result_root/root.pdf` とする。

## 6. 変換パイプライン設計
1. CLI引数の解析（commander）
2. 設定ファイル読込（任意）
3. 設定マージ（default < config < CLI）
4. 入力解決（対象Markdown列挙）
5. Markdown→HTML変換（markdown-it + markdown-it-mermaid）
6. HTMLテンプレート合成（CSS/mermaid初期化スクリプト埋込）
7. ヘッドレスブラウザでHTMLロード
8. Mermaid描画完了待機
9. PDF出力
10. 一時ファイル削除（debug時は保持）

## 7. Mermaid描画完了判定
- 判定条件:
  - Mermaid初期化後、対象ノードがSVGとして描画済み
  - 追加で短時間のDOM安定を確認
- タイムアウト時:
  - 対象ファイルを失敗として記録
  - `--debug` 有効時は中間HTML等を保存

## 8. エラー処理方針（確定）

### 8.1 フォルダ一括変換時
- 1ファイル失敗で全体停止はしない（継続処理）。
- 最終結果で成功件数・失敗件数・失敗一覧を返す。

### 8.2 終了コード
- `0`: 全件成功
- `1`: 一部失敗（部分成功）
- `2`: 引数/入力バリデーションエラー
- `3`: 変換エラー（Markdown/HTML生成）
- `4`: Mermaid描画タイムアウト
- `5`: PDF出力エラー
- `9`: 想定外エラー

## 9. ログ/レポート
- 標準出力:
  - 対象件数
  - 成功件数
  - 失敗件数
- 失敗時:
  - ファイル単位の失敗理由を一覧表示
- `--debug`:
  - 中間HTML
  - ブラウザレンダリング関連ログ

追加CR:
- `--report-format json` 時は、標準出力を単一JSONドキュメントに固定し、進捗ログは標準エラー出力へ分離する。
- JSONレポートには `schemaVersion` を付与する（初版: `1.0`）。

## 10. アーキテクチャ
- `cli` 層: 引数定義・バリデーション
- `config` 層: 設定読込・マージ
- `resolver` 層: 入力探索・出力先解決
- `renderer` 層: Markdown/HTML変換・Mermaid初期化
- `pdf` 層: PDF生成バックエンド
- `report` 層: 成否集計と結果整形

補足:
- PDF生成バックエンドは将来差し替え可能な抽象を設ける（headless / electron等）。

## 11. PDF生成方式比較（検討結果）

### 11.1 ヘッドレスブラウザ（初版採用）
利点:
- CLI用途に最短で適合
- Mermaid描画後PDF化の実装実績が多い
- CI自動化しやすい

課題:
- Chromium依存の環境準備
- フォント差異調整が必要

### 11.2 Electron
利点:
- Chromium同梱で描画再現性が高い
- `printToPDF` 利用で品質安定
- 将来GUI化に発展しやすい

課題:
- 配布サイズ・起動コストが大きい
- CLI単体には依存が重い

### 11.3 Tauri
利点:
- 軽量配布が期待できる
- デスクトップ製品化時に有力

課題:
- PDF経路の実装複雑度が上がりやすい
- WebView差異の吸収コスト

### 11.4 採用提案
- 初版: ヘッドレスブラウザ
- 将来GUI要求が確定した時点でElectron/Tauriを再評価

## 12. 受け入れ条件（設計版）
1. `--input` にファイル/フォルダを指定できる。
2. input=フォルダ時、再帰的に `*.md` を対象化できる。
3. `--output` にファイル/フォルダを指定できる。
4. 複数入力時、出力先に入力相対フォルダ構造を再現して出力できる。
5. 一括変換中に失敗があっても継続し、最終で失敗一覧を返せる。
6. Mermaid記法を含むMarkdownをPDF化できる。
7. 初版のPDFバックエンドはヘッドレスブラウザを採用している。
8. `--help-format json` で機械可読なヘルプを出力できる。
9. `--report-format json` で機械可読な実行結果を出力できる。
10. `--plan` で変換なしの実行計画（入力/出力マップ）を取得できる。
11. JSONレポートに `schemaVersion` が含まれる。
12. 失敗時に固定語彙の失敗種別コードを出力できる。

## 13. クローズドクエッション結果
- 同名ファイル衝突対応:
  - 連番ではなく、入力相対フォルダ構造を出力先へ再現する方針で確定。
- 一括変換時の失敗ポリシー:
  - 継続処理し、最後に失敗一覧を返す方針で確定。
- 再帰探索の既定値:
  - ONで確定。

## 15. 変更要求（CR）: AIエージェント利用性

### 15.1 背景
- 本CLIはAIエージェントから自動実行される前提がある。
- 人間中心のhelp/logのみでは、エージェントが誤解釈しやすい。

### 15.2 変更要求
1. 引数仕様の機械可読化（`--help-format json`）
2. 実行結果の機械可読化（`--report-format json`）
3. 事前検証モード（`--plan`）
4. エラー分類の固定語彙化
5. JSONスキーマバージョン管理

### 15.3 期待効果
- エージェントが「解析 -> 計画 -> 実行 -> 検証」を安定して自動化できる。
- CIや他ツール連携時のパース失敗を減らせる。
- 将来のCLI拡張時も、互換性管理を明示できる。
